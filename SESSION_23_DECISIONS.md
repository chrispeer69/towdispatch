# Session 23 — Lien Processing: Decisions Log

**Scope:** Statutory lien-sale workflow for unclaimed impounded vehicles across
the top 10 states (CA, TX, FL, NY, GA, NC, OH, IL, PA, MI), with a pure rule
engine, NestJS API, env-gated nightly cron, shared Zod contracts, web UI, and
state-form PDF rendering.

---

## ⚠️ LEGAL DISCLAIMER (read first)

The per-state day-counts, value thresholds, publication rules, and statute
citations in this module are **best-effort interpretations** researched against
best available knowledge. **They are not legal advice and have NOT been verified
against the current state codes.** Every per-state config block carries a
`// Statute: … — verify against current state code before production use` note.

**Legal counsel review is required before any production lien sale runs through
this code.** The module is deliberately built so a human makes every legal
decision (see "Observation-only cron" and "No auto-advance" below).

## Per-state rules (statute cited · conservative choice · why)

Runtime source of truth: `apps/api/src/modules/lien-processing/state-rules.config.ts`.
Mirrored (seeded) into `lien_state_rules` by `0037_lien_processing.sql`.

| CA | Civil §3068.1 / Veh §22851.12, §22851.10 | 3 | 10 / 10 | yes (+10) | 30 | yes (< $4,000) |
| TX | Occ. Code 2303 / Prop. §70.006 | 5 | 30 / 30 | **no** (certified) | 30 | n/a |

| MI | MCL §257.252 / §570.521-.530 | 7 | 20 / 20 | yes (+10) | 30 | yes |

**Conservative-vs-aggressive choices:**
- **Floor on the sale date.** The earliest legal sale date is the *max* of the
  minimum holding period AND every applicable notice/publication waiting window
  — never the min. A case can only become sellable once all of them have
  elapsed (conservative).
- **Unknown value → `mid` tier.** When the operator hasn't recorded an estimated
  value, the case is treated as `mid`, which keeps the publication requirement
  on. We never silently drop to the low-value (publication-exempt) path.
- **Owner not located → publication required** regardless of value tier, in any
  state that has a publication mechanism. Publication substitutes for personal
  notice; we'd rather over-notice than under-notice.
- **FL low-value is NOT publication-exempt.** FL §713.78 does not carve out a
  low-value exemption in the way most states do, so `lowValuePublicationExempt`
  is `false` for FL only.
- **TX & OH have no publication step.** Both rely on certified notice to the
  last-known owner + the waiting period; `publicationRequired: false`. Owner-
  not-found there proceeds on certified notice + the waiting period.

These values are intentionally simple and internally consistent so the engine
is fully unit-tested per state; the exact statutory numbers are the part that
needs counsel sign-off.

## No auto-advance / observation-only cron

- The service **never decides to sell.** Every legal step — request DMV lookup,
  record a notice, record a response, mark ready-for-sale, record a sale — is an
  explicit operator action. `advanceCase` only moves the workflow forward when
  the prerequisite has been satisfied (e.g. it refuses to skip a required notice
  and returns 409).
- A **sale (`closeCase` disposition `sold`) is gated on `status = ready_for_sale`** —
  the statutory prerequisites must have been satisfied first.
- The nightly cron (`LIEN_ADVANCE_CRON_ENABLED`, **default false**, `0 3 * * *`,
  one hour after the impound accrual cron) is **observation-only**: it recomputes
  `next_action_due_at` and appends a single `action_due` timeline event per
  (case, due-date) for overdue cases. It NEVER sends a notice, advances a step,
  or marks a case ready/sold. Auto-progressing a statutory lien sale is exactly
  the kind of automation that draws regulator complaints.

## PDF template source — generated, one renderer

- **No official state PDF form was sourced this session.** Each notice is a
  **compliant text document** generated by a single PDFKit renderer
  (`forms/lien-form.renderer.ts`) that cites the governing statute and states
  the vehicle, the charges owed, the redemption right, and the earliest sale
  date.
- "20 templates" = 10 states × 2 form types (owner notice + publication notice)
  as **logical** templates, all produced by one renderer driven by the per-state
  rule config — not 20 physical files. One renderer is the same coverage with a
  fraction of the maintenance surface; sourcing the official fillable forms is
  deferred.
- **Documents are English-only** (statutory filing language). A bilingual
  (EN/ES) courtesy line is included in the redemption-rights block so the
  recipient understands their right to reclaim, but the legal body stays English
  for filing accuracy. This is a deliberate exception to the general Spanish-
  parity rule for user-facing strings.

## "Open lien case from impound record" without touching the impound module

- The task DO-NOT list includes the impound module; the web deliverable asks for
  an open-case action from an impound record. Resolved by **not editing any
  impound file**: `/lien-cases/new` reads lien-eligible records from the existing
  `GET /impound/records?lienEligible=true` endpoint (read-only consumption) and
  accepts `?impoundRecordId=` to preselect. No impound API, schema, or web file
  was modified.

## Other decisions

- **One live lien case per impound record** — partial unique index on
  `impound_record_id WHERE deleted_at IS NULL`; a duplicate open returns 409.
- **Pending-notice idempotency** — partial unique on
  `(lien_case_id, notice_type, recipient_role) WHERE response_received_at IS NULL
  AND deleted_at IS NULL`. One unanswered notice of a given type/recipient at a
  time; recording the response frees a follow-up.
- **`lien_state_rules` is global reference data** — no `tenant_id`, no RLS
  (statutory rules are identical for every operator in a state). `app_user` reads
  it via the default-privilege grant. The runtime engine uses the typed config;
  the table mirrors it for queryability and future tenant overrides.
- **State-rules endpoint serves the typed config** (`GET /lien-cases/state-rules`)
  rather than the DB row — the config is the source of truth and is always
  available, even before the seed runs.
- **Migration numbering**: `0037_lien_processing.sql`. Two pre-existing `0036_*`
  files (impound + onboarding) are parallel-session artifacts; `0037` is unique.
  `scripts/check-migrations.sh` (pre-existing `0034` dupe) was not touched.
